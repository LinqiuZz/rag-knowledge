# 📄 RAG系统文档分块策略详解

## 分块策略概述

RAG知识库系统提供了三种不同的文档分块策略，可以根据文档类型和使用场景灵活选择：

1. **递归分块 (recursive)** - 默认策略，按层级分隔符递归切分
2. **句子分块 (sentence)** - 保持句子完整性，适合技术文档
3. **段落分块 (paragraph)** - 保持段落完整性，适合结构化文档

## 配置参数

在 `config.yaml` 中定义了分块的默认参数：

```yaml
chunking:
  size: 512        # 每块目标字符数
  overlap: 64      # 块间重叠字符数
  strategy: sentence  # 分块策略: recursive | sentence | paragraph
```

- **chunk_size**: 控制每个文本块的最大字符数，建议范围256-1024
- **overlap**: 控制相邻文本块之间的重叠字符数，建议为chunk_size的10%-15%
- **strategy**: 选择合适的分块策略

## 各种分块策略详解

### 1. 递归分块 (recursive)

递归分块是系统默认的分块策略，采用多层次的分隔符优先级系统：

```python

    separators = ["\n\n", "\n", "。", "！", "？", ".", "!", "?", "；", ";", " "]
    # 按优先级顺序尝试分隔符
```

**分块优先级**：
1. 按标题/章节切分（Markdown标题、数字编号）
2. 按段落切分（双换行 `\n\n`）
3. 按句子切分（中文句号、问号、感叹号）
4. 按换行切分（单换行 `\n`）
5. 按标点符号切分（分号、逗号等）
6. 按空格切分
7. 硬切（最后手段）

**优点**：能够很好地保持文档结构和语义完整性
**缺点**：可能产生不均匀的块大小
**适用场景**：通用文档、混合格式文档

### 2. 句子分块 (sentence)

句子分块策略专注于保持句子的完整性：

```python
def _sentence_split(text: str, chunk_size: int, overlap: int) -> list[str]:
    # 中英文句子分隔符
    sentences = re.split(r"(?<=[。！？.!?])\s*", text)
    sentences = [s.strip() for s in sentences if s.strip()]
```

**优点**：
- 保持句子完整性，避免截断语义
- 对于问答系统特别有用
- 保持语法结构

**缺点**：
- 如果有很长的句子，仍需要回退到递归分块
- 可能导致某些块超出设定大小

**适用场景**：
- 技术文档
- 法律文本
- 学术论文
- 需要精确语义的场景

### 3. 段落分块 (paragraph)

段落分块策略专注于保持段落的完整性：

```python
def _paragraph_split(text: str, chunk_size: int, overlap: int) -> list[str]:
    paragraphs = text.split("\n\n")
    paragraphs = [p.strip() for p in paragraphs if p.strip()]
```

**优点**：
- 保持段落的完整性和上下文
- 适合需要保持连贯性的文档
- 通常有更好的阅读体验

**缺点**：
- 某些段落可能很长，需要回退到句子或递归分块
- 可能产生不均匀的块大小

**适用场景**：
- 小说、文章
- 结构化文档
- 需要保持上下文连贯性的文档

## 实际分块过程

### 预处理阶段
1. 规范化换行符 (`\r\n` → `\n`)
2. 清理多余空白字符

### 分块阶段
1. 优先按标题切分（Markdown标题或数字编号）
2. 对每个章节应用选定的分块策略
3. 处理超长内容（回退到更细粒度的分块）

### 后处理阶段
1. 应用块间重叠，保证上下文连续性
2. 合并过小的块，避免碎片化
3. 生成块ID和元数据

## 分块策略选择建议

| 文档类型 | 推荐策略 | 理由 |
|----------|----------|------|
| 技术文档 | sentence | 保持句子完整性，便于理解技术概念 |
| 学术论文 | sentence | 保持实验步骤、公式说明的完整性 |
| 小说文章 | paragraph | 保持段落连贯性，提升阅读体验 |
| 网页内容 | recursive | 多样化内容，递归分块适应性强 |
| 代码文档 | sentence | 保持代码片段和说明的完整性 |
| 法律文书 | sentence | 保持条款和解释的完整性 |

## 性能考虑

### 块大小影响
- **较小块（256-384字符）**：
  - 优点：精确匹配，内存占用低
  - 缺点：增加查询数量，可能丢失上下文
  
- **较大块（768-1024字符）**：
  - 优点：保留更多上下文，减少查询次数
  - 缺点：可能包含无关信息，内存占用高

### 重叠策略
- 重叠有助于查询时保持上下文连续性
- 过大的重叠会增加存储和计算开销
- 建议重叠为块大小的10%-15%
